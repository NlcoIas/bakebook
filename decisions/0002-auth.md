# ADR 0002: Authentication via Cloudflare Access

## Status
Accepted

## Context
Bakebook is a single-user application. We need authentication to prevent unauthorized access but don't need user management, registration, or multi-tenancy.

## Decision
Use Cloudflare Access as the sole authentication perimeter in v1:
- Cloudflare Access policy restricts `bakebook.nicolasschaerer.ch` to a single email address
- The API reads `Cf-Access-Authenticated-User-Email` header and trusts it
- In dev mode (`ENV=dev`), the API injects `dev@local` as the authenticated user
- No app-level auth (no JWT, no sessions, no cookies managed by the app)

## Security Model
- Cloudflare Tunnel ensures traffic only reaches the app through Cloudflare's edge
- Coolify network policy restricts API ingress to the web service + tunnel only
- The `Cf-Access-Authenticated-User-Email` header is set by Cloudflare and cannot be spoofed by end users (Cloudflare strips any user-supplied values)

## Consequences
- Zero auth code to maintain in the app
- Adding a second user in the future would require either expanding the Access policy or adding app-level auth
- Dev environment has no auth friction — just works
- If Cloudflare Access has an outage, the app is inaccessible (acceptable for a personal tool)
